# How the FastAPI app connects to Postgres in Docker

Your app does not open database connections by itself: **something in your process environment must provide `DATABASE_URL`**. Postgres running in Docker is reachable at **different hostnames and ports** depending on whether the **Python process runs on your machine** or **inside the same Compose network** as the database.

See also: [API_DOCKER_GUIDE.md](../API_DOCKER_GUIDE.md) (full stack), [POSTGRES_DOCKER_GUIDE.md](../POSTGRES_DOCKER_GUIDE.md) (Postgres container and ports).

---

## What the app uses

- **One setting:** `DATABASE_URL` — a single string, typically read with **Pydantic Settings** (you already have `pydantic-settings` and `python-dotenv` in [requirements.txt](../requirements.txt)).
- **Driver in the URL:** this project’s Compose file uses the **Psycopg 3** SQLAlchemy dialect:

  `postgresql+psycopg://USER:PASSWORD@HOST:PORT/DATABASE`

  When you implement the DB layer, add a Psycopg 3 package (for example `psycopg[binary]`) alongside [SQLAlchemy](../requirements.txt) so that URL scheme works.

---

## Case A — API runs **inside** Docker Compose (`api` service)

[docker-compose.yaml](../docker-compose.yaml) injects the URL for you:

```yaml
DATABASE_URL: postgresql+psycopg://${POSTGRES_USER}:${POSTGRES_PASSWORD}@database:5432/${POSTGRES_DB}
```

- **`database`** is the **Compose service name**. On the Compose network, that name resolves to the Postgres container. You use the **container** port **`5432`**, not `POSTGRES_PUBLISH_PORT`.
- **`depends_on`** only starts containers in order; it does not wait until Postgres accepts connections. For robust startup, add retries or a healthcheck (see [API_DOCKER_GUIDE.md](../API_DOCKER_GUIDE.md)).

Your code should read `os.environ["DATABASE_URL"]` or `Settings().database_url` and pass it to SQLAlchemy’s `create_engine(...)`.

---

## Case B — API runs **on your machine** (venv + `uvicorn` / `fastapi dev`)

Postgres is still in Docker, but your process is **not** on the Compose service network. It reaches Postgres via the **published** port on the host (from [.env.example](../.env.example): `POSTGRES_PUBLISH_PORT`, often `5435`).

Example `.env` on the host:

```env
DATABASE_URL=postgresql+psycopg://postgres:password@localhost:5435/fastapi_crud
```

Use **`localhost`** (or `127.0.0.1`) and the **host** port from `ports:` in compose (`"${POSTGRES_PUBLISH_PORT}:5432"`). Do **not** use hostname `database` from the host unless you have extra DNS setup.

## Minimal wiring pattern (when you add a DB layer)

The important part for Docker is **not** the Python API shape—it is that **`DATABASE_URL` matches where the process runs** (Compose service name + `5432` vs `localhost` + publish port).

The steps below are a **guide only**: create or adjust files to match how you want the project structured. Paths assume your package is `app/` at the repo root (same as today).

### 0. Dependency

Add a Psycopg 3 build to your environment (not in [requirements.txt](../requirements.txt) yet unless you add it):

```bash
pip install "psycopg[binary]"
pip freeze > requirements.txt
```

Pin a version in `requirements.txt` when you are happy with it, e.g. `psycopg[binary]==3.x.x`.

### 1. Settings — load `DATABASE_URL`

Example **`app/config.py`**:

```python
from pathlib import Path
from pydantic_settings import BaseSettings, SettingsConfigDict

_BASE_DIR = Path(__file__).resolve().parent.parent


class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=_BASE_DIR / ".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )
    database_url: str  # from env DATABASE_URL

    # If not database_url
    # database_url: str = Field(validation_alias="CUSTOM_DATABASE_URL")


# Env / .env supply fields; pyright cannot see that at call time.
def get_settings() -> Settings:
    return Settings()  # type: ignore[call-arg]

```

- For **Case B**, keep `DATABASE_URL` in `.env` at the project root (or export it in the shell). `env_file=".env"` is resolved relative to the **current working directory** when you start uvicorn from the repo root (`uvicorn app.main:app`).
- For **Case A**, Compose sets `DATABASE_URL` in the container environment; you do not need a `.env` inside the image unless you choose to copy one in.

### 2. Engine and sessions — one engine per process

Example **`app/db.py`**:

```python
from collections.abc import Generator

from sqlalchemy import create_engine
from sqlalchemy.orm import Session, sessionmaker

from app.config import settings

engine = create_engine(
    settings.database_url,
    pool_pre_ping=True,
)

SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)


def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

`pool_pre_ping=True` helps when the DB container restarts: the pool checks connections before use.

### 3. Use the session in a route

Example snippet for **`app/main.py`** (illustration — merge with your existing app):

```python
from fastapi import Depends, FastAPI
from sqlalchemy import text
from sqlalchemy.orm import Session

from app.db import get_db

app = FastAPI()


@app.get("/db-ping")
def db_ping(db: Session = Depends(get_db)):
    one = db.execute(text("SELECT 1")).scalar_one()
    return {"ok": True, "select_1": one}
```

### 3a. Test the database connection

1. **Rebuild your services** using Docker Compose:

```bash
docker compose stop api
docker compose up -d --build api
```

2. **Open Swagger UI** in your browser at [http://localhost:API_PUBLISH_PORT/docs](http://localhost:API_PUBLISH_PORT/docs)
   - Replace `API_PUBLISH_PORT` with the value set in your `.env` file (e.g., `8000`, `8016`, etc.).

3. **Try the `/db-ping` endpoint**:
   - Click on `/db-ping`, then "Try it out" and "Execute".
   - If everything is configured correctly and Postgres is running, you should see a response like:
     ```json
     {
       "ok": true,
       "select_1": 1
     }
     ```
   - If your `DATABASE_URL` is incorrect, or if Postgres is not running, you will see an error message in the response or in the API logs.

Tip: You can also test the endpoint from your terminal or command prompt:

**Linux / macOS:**

```
curl http://localhost:API_PUBLISH_PORT/db-ping
```

**Windows (Command Prompt or PowerShell):**

```
curl http://localhost:API_PUBLISH_PORT/db-ping
```

(Again, use the correct port from your `.env`.)

### 4. Optional: dispose engine on shutdown

If you use a **lifespan** context on `FastAPI`, you can call `engine.dispose()` on shutdown to close pooled connections cleanly. For a small dev app, omitting this is often fine.

### 5. Models and tables (next step)

When you add ORM models, use SQLAlchemy 2.0 style (`DeclarativeBase`, `Mapped`, `mapped_column`), run migrations (e.g. Alembic), or call `Base.metadata.create_all(engine)` once in dev — that is separate from “can I connect?”.

## Quick checklist

| App runs where          | `DATABASE_URL` host | Port in URL             |
| ----------------------- | ------------------- | ----------------------- |
| Compose `api` container | `database`          | `5432`                  |
| Your laptop (venv)      | `localhost`         | `POSTGRES_PUBLISH_PORT` |

If authentication fails after you change `POSTGRES_USER` / `POSTGRES_PASSWORD` in `.env`, the data directory may have been initialized with old credentials—see **Reset dev data** in [POSTGRES_DOCKER_GUIDE.md](../POSTGRES_DOCKER_GUIDE.md).
