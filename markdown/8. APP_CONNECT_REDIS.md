# How the FastAPI app connects to Redis in Docker

Your app does not open Redis connections by itself: **something in your process environment must provide `REDIS_URL`**. Redis running in Docker is reachable at **different hostnames and ports** depending on whether the **Python process runs on your machine** or **inside the same Compose network** as Redis.

See also: [API_DOCKER_GUIDE.md](../API_DOCKER_GUIDE.md) (full stack), [REDIS_DOCKER_GUIDE.md](../REDIS_DOCKER_GUIDE.md) (Redis container and ports).

---

## What the app uses

- **One setting:** `REDIS_URL` — a single string, typically read with **Pydantic Settings** (you already have `pydantic-settings` and `python-dotenv` in [requirements.txt](../requirements.txt)).
- **URL shape:** this project’s Compose file uses **password authentication** (`--requirepass` on the Redis server). The usual form is:

  `redis://:PASSWORD@HOST:PORT/DATABASE_INDEX`

  The **colon before `PASSWORD`** means “no username” (Redis ACLs can use a username in other setups; this stack uses the classic password-only pattern).

- **Client library:** use the official **`redis`** package (redis-py) with `Redis.from_url(...)` or `redis.from_url(...)`.

---

## Case A — API runs **inside** Docker Compose (`api` service)

[docker-compose.yaml](../docker-compose.yaml) injects the URL for you:

```yaml
REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
```

- **`redis`** is the **Compose service name**. On the Compose network, that name resolves to the Redis container. You use the **container** port **`6379`**, not `REDIS_PUBLISH_PORT`.
- **`depends_on`** only starts containers in order; it does not wait until Redis accepts commands. For robust startup, add retries in app code or a healthcheck (see [API_DOCKER_GUIDE.md](../API_DOCKER_GUIDE.md)).

Your code should read `os.environ["REDIS_URL"]` or `Settings().redis_url` and pass it to `redis.from_url(...)`.

---

## Case B — API runs **on your machine** (venv + `uvicorn` / `fastapi dev`)

Redis is still in Docker, but your process is **not** on the Compose service network. It reaches Redis via the **published** port on the host (from [.env.example](../.env.example): `REDIS_PUBLISH_PORT`, often `6380`).

Example `.env` on the host:

```env
REDIS_PASSWORD=sample_password
REDIS_URL=redis://:sample_password@localhost:6380/0
```

Use **`localhost`** (or `127.0.0.1`) and the **host** port from `ports:` in compose (`"${REDIS_PUBLISH_PORT}:6379"`). Do **not** use hostname `redis` from the host unless you have extra DNS setup.

If the password contains characters that are not safe in URLs (`@`, `:`, `/`, `#`, etc.), **URL-encode** the password in `REDIS_URL`.

## Minimal wiring pattern (when you add a Redis layer)

The important part for Docker is **not** the Python API shape—it is that **`REDIS_URL` matches where the process runs** (Compose service name + `6379` vs `localhost` + publish port).

The steps below are a **guide only**: create or adjust files to match how you want the project structured. Paths assume your package is `app/` at the repo root (same as today).

### 0. Dependency

Add **redis-py** to your environment (not in [requirements.txt](../requirements.txt) until you add it):

```bash
pip install redis
pip freeze > requirements.txt
```

Pin a version in `requirements.txt` when you are happy with it, e.g. `redis==5.x.x`.

### 1. Settings — load `REDIS_URL`

Extend **`app/config.py`** with a field for Redis (example — merge with your existing `Settings`):

```python
class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=_BASE_DIR / ".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )
    database_url: str  # from env DATABASE_URL
    redis_url: str  # from env REDIS_URL
```

- For **Case B**, keep `REDIS_URL` (and `REDIS_PASSWORD` if you build the URL yourself) in `.env` at the project root.
- For **Case A**, Compose sets `REDIS_URL` in the container environment; you do not need a `.env` inside the image unless you choose to copy one in.

Keep the same `settings = Settings()  # type: ignore[call-arg]` pattern you use for `database_url`.

### 2. Client factory — one pattern per process

Example **`app/redis_client.py`** (name as you prefer):

```python
from collections.abc import Generator

import redis

from app.config import settings


def get_redis() -> Generator[redis.Redis, None, None]:
    client = redis.from_url(settings.redis_url, decode_responses=True)
    try:
        yield client
    finally:
        client.close()
```

Here is a sample of how to use a shared Redis connection pool, which is recommended for high-traffic applications:

```python
import redis
from app.config import settings

# Create a shared connection pool at import time
redis_pool = redis.ConnectionPool.from_url(settings.redis_url, decode_responses=True)
redis_client = redis.Redis(connection_pool=redis_pool)

def get_redis():
    # Return the shared client for each request
    return redis_client
```

You would typically use this shared pool approach in scenarios where your FastAPI application expects moderate to heavy Redis traffic (such as caching, rate limiting, or background task processing), to improve efficiency and resource management. Specifically, use this pattern if:

- Your app serves many concurrent requests, and you want to minimize connection overhead between requests.
- Your Redis usage involves multiple endpoints or components that should all use the same connection pool.
- You are deploying to production or a load-tested environment with expected parallel load.

Use `get_redis` as your FastAPI dependency, replacing any pattern where you previously created a new Redis client per request. For example, in your route:

```python
from fastapi import Depends

@app.get("/any-endpoint")
def endpoint(r: redis.Redis = Depends(get_redis)):
    # use r for any redis operation, benefiting from the shared connection pool
    ...
```

This setup ensures all threads/workers in your app share a single connection pool, reducing connection churn and maximizing Redis efficiency under load.

### 3. Use the client in a route

Example snippet for **`app/main.py`** (illustration — merge with your existing app):

```python
from collections.abc import Callable
from typing import cast

from fastapi import Depends, FastAPI
import redis

from app.redis_client import get_redis

app = FastAPI()


@app.get("/redis-ping")
def redis_ping(r: redis.Redis = Depends(get_redis)):
    # Same Depends(get_redis) pattern as in app/redis_client.py.
    pong = cast(Callable[[], bool], r.ping)()
    return {"ok": True, "ping": pong}
```

`PING` returns `True` when Redis answers (with `decode_responses=True`, redis-py returns a boolean for `PING`). With **strict** type checking, redis-py’s stubs type `ping` in a way that triggers “partially unknown”; the `cast` documents a normal synchronous `ping()` call.

### 3a. Test the Redis connection

1. **Rebuild your services** using Docker Compose:

```bash
docker compose stop api
docker compose up -d --build api
```

2. **Open Swagger UI** in your browser at [http://localhost:API_PUBLISH_PORT/docs](http://localhost:API_PUBLISH_PORT/docs)
   - Replace `API_PUBLISH_PORT` with the value set in your `.env` file (e.g., `8000`, `8016`, etc.).

3. **Try the `/redis-ping` endpoint**:
   - Click on `/redis-ping`, then "Try it out" and "Execute".
   - If everything is configured correctly and Redis is running, you should see a response like:
     ```json
     {
       "ok": true,
       "ping": true
     }
     ```
   - If your `REDIS_URL` is wrong, the password does not match `--requirepass`, or Redis is not running, you will see an error in the response or in the API logs.

Tip: You can also test the endpoint from your terminal or command prompt:

**Linux / macOS:**

```
curl http://localhost:API_PUBLISH_PORT/redis-ping
```

**Windows (Command Prompt or PowerShell):**

```
curl http://localhost:API_PUBLISH_PORT/redis-ping
```

(Again, use the correct port from your `.env`.)

You can also run **`redis-cli`** against the published port (see [REDIS_DOCKER_GUIDE.md](../REDIS_DOCKER_GUIDE.md)) to confirm Redis itself is up outside the API.

### 4. Optional: close shared pools on shutdown

If you switch to a **module-level** `redis.Redis` or `ConnectionPool`, register a **lifespan** on `FastAPI` and close the pool on shutdown. For a small dev app, omitting this is often fine.

### 5. Usage beyond ping (next step)

Caches, rate limits, sessions, and task queues are all “what you do **after** you can connect.” They are separate from “can I reach Redis?”—pick patterns (TTL keys, hash structures, streams, etc.) to match your feature.

## Quick checklist

| App runs where          | `REDIS_URL` host | Port in URL             |
| ----------------------- | ---------------- | ----------------------- |
| Compose `api` container | `redis`          | `6379`                  |
| Your laptop (venv)      | `localhost`      | `REDIS_PUBLISH_PORT`    |

If authentication fails after you change `REDIS_PASSWORD` in `.env`, ensure **Compose**, **`.env` for local uvicorn**, and **any persisted volume** still match—see password and persistence notes in [REDIS_DOCKER_GUIDE.md](../REDIS_DOCKER_GUIDE.md).
