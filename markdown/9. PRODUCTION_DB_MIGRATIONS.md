# Production: migrate database tables (Alembic)

In **development**, this project can create missing tables with **`Base.metadata.create_all`** (see `init_development_tables()` in [app/db.py](../app/db.py) and the lifespan hook in [app/main.py](../app/main.py)). That is convenient but **not** how you should manage schema in **production**.

**Why migrations in production**

- You need a **repeatable, reviewed history** of changes (add column, rename, indexes), not “whatever the models happen to imply today.”
- **`create_all`** does **not** drop columns, rename tables safely, or migrate data—it only creates missing objects.
- **Alembic** is the usual companion to **SQLAlchemy** for versioned schema upgrades (and controlled rollbacks when you write `downgrade` steps).

**This guide** assumes Postgres and the same **`DATABASE_URL`** shape as [.env.example](../.env.example) (`postgresql+psycopg://...`).

---

## 1. Prerequisites

- **SQLAlchemy models** use a single **`DeclarativeBase`** (this repo: `Base` in [app/features/crud/models/items_model.py](../app/features/crud/models/items_model.py)).
- **Production** runs with **`ENVIRONMENT=production`** (or any value other than `development`) so **`create_all` does not run** on app startup—only Alembic should change the live schema.
- **Backup** the database before running migrations against real data.

---

## 2. Add Alembic to the project (one-time)

From the **repository root**, with your virtualenv active:

```bash
pip install alembic
pip freeze > requirements.txt
```

Generate the Alembic layout (creates an `alembic/` folder and `alembic.ini`):

```bash
alembic init alembic
```

You will edit **`alembic/env.py`** and usually trim **`sqlalchemy.url`** out of **`alembic.ini`** so the URL always comes from **`DATABASE_URL`** (see below).

---

## 3. Point Alembic at your models and `DATABASE_URL`

### `alembic.ini`

- Set **`script_location = alembic`** (default after `alembic init`).
- **Comment out** the default **`sqlalchemy.url = driver://user:pass@localhost/dbname`** line. If **`env.py`** still used **`engine_from_config`**, that placeholder would make SQLAlchemy look for a dialect named **`driver`** and fail. This repo’s **`env.py`** ignores that URL and uses **`DATABASE_URL`** from **`get_settings()`** instead.

### `alembic/env.py`

After `alembic init`, **replace** the generated **`alembic/env.py`** with something like this project’s version. The stock template uses **`engine_from_config`** and reads **`sqlalchemy.url = driver://...`** from **`alembic.ini`**, which triggers **`NoSuchModuleError: Can't load plugin: sqlalchemy.dialects:driver`**. Here we use **`create_engine(get_settings().database_url)`** instead (same URL as the FastAPI app, from `.env` / `DATABASE_URL`).

This file should:

1. Load **`DATABASE_URL`** via [app/config.py](../app/config.py) (`get_settings()`).
2. Set **`target_metadata = Base.metadata`** so **autogenerate** can compare models to the database.

**Important:** import **`Base`** from your models module (and import any other model modules that use the same `Base` so their tables are registered on `metadata` before autogenerate runs).

**Replace the default `alembic/env.py` with:**

```python
from logging.config import fileConfig

from alembic import context
from sqlalchemy import create_engine, pool

from app.config import get_settings
from app.features.crud.models.items_model import Base

config = context.config

if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata


def run_migrations_offline() -> None:
    url = get_settings().database_url
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
    )

    with context.begin_transaction():
        context.run_migrations()


def run_migrations_online() -> None:
    connectable = create_engine(
        get_settings().database_url,
        poolclass=pool.NullPool,
    )

    with connectable.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
        )

        with context.begin_transaction():
            context.run_migrations()


if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

Optional: add imports under `Base` for other apps, e.g. `# from app.features.orders.models.order_model import Order  # noqa: F401`.

**Do not** rely on **`engine_from_config`** + the placeholder **`driver://`** URL in **`alembic.ini`** for this project; comment that line out in **`alembic.ini`** (see note in the next subsection if you add one there).

## Docker Compose: set up migrations and what to change

Use this when you run Postgres + API with **[docker-compose.yaml](../docker-compose.yaml)** and want **`alembic upgrade head`** to work **inside a container** (same DB the **`api`** service uses: host **`database`**, port **`5432`**).

### What you get

- The **API image** includes **`app/`**, **`alembic/`**, and **`alembic.ini`** so one-off containers can run Alembic.
- A **`migrate`** service (Compose **profile** `migrate`) runs **`alembic upgrade head`** and exits; it is **not** started by plain **`docker compose up`** so migrations stay manual and predictable.

### 1. One-time project steps (before Docker)

1. Add **`alembic`** to **[requirements.txt](../requirements.txt)** (already in this repo).
2. Finish **§2–§3**: **`alembic init`**, fix **`env.py`** and **`alembic.ini`** so **`DATABASE_URL`** comes from **`get_settings()`** (see [alembic/env.py](../alembic/env.py)).

Without that, the container has nothing useful to run.

### 2. `Dockerfile` — files to copy and `PYTHONPATH`

The image **`WORKDIR`** is **`/app`**. The FastAPI package lives at **`/app/app`**. Alembic must see **`alembic.ini`** at **`/app/alembic.ini`** and import **`app.config`**, so the [Dockerfile](../Dockerfile) should include:

| Change                       | Why                                                                                                                           |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **`COPY alembic.ini .`**     | Alembic’s CLI looks for config next to the working directory / project root.                                                  |
| **`COPY alembic ./alembic`** | Revision scripts and **`env.py`** must be in the image.                                                                       |
| **`ENV PYTHONPATH=/app`**    | So **`from app.config import get_settings`** works when you run **`alembic`** (same as Uvicorn resolving **`app.main:app`**). |

If you add new migration-only dependencies later, update **`requirements.txt`** and **rebuild** the image.

**Rebuild** after changing Dockerfile lines:

```bash
docker compose build --no-cache api
```

(`migrate` uses the **same** `build: .`, so one build covers both.)

### 3. `docker-compose.yaml` — shared env and the `migrate` service

This repo uses a YAML anchor so **`api`** and **`migrate`** get the **same** variables **[app/config.py](../app/config.py)** needs (not only `DATABASE_URL`).

| Piece                   | What to do                                                                                                                                                                                                                                                                                                                            |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`x-api-environment`** | Lists env vars injected into containers: in-network **`DATABASE_URL`** (`database:5432`), **`REDIS_URL`** (`redis:6379`), **`APP_NAME`**, **`JWT_*`**, etc. Compose substitutes **`${POSTGRES_USER}`** and friends from your **host** `.env` (see [.env.example](../.env.example)).                                                   |
| **`api`**               | **`environment: <<: *api-environment`** — normal web process; **`CMD`** is Uvicorn in the Dockerfile.                                                                                                                                                                                                                                 |
| **`migrate`**           | **`profiles: ["migrate"]`** — only runs when you opt in. **`command: python -m alembic upgrade head`** (use **`python -m alembic`** in containers so Alembic runs even if the `alembic` script is not on **`PATH`**). **`depends_on: database`** so Postgres is started first (see [6. API_DOCKER_GUIDE.md](6.%20API_DOCKER_GUIDE.md): **`depends_on` is not a readiness guarantee**; if the DB is slow to accept connections, add retries or a healthcheck). |

**If you add a new required field to `Settings`**, update **three places** so Docker stays in sync:

1. **[app/config.py](../app/config.py)** — new field.
2. **[.env.example](../.env.example)** — document the variable.
3. **`x-api-environment` in [docker-compose.yaml](../docker-compose.yaml)** — pass the value into **`api`** and **`migrate`** (usually **`${VAR:-default}`**).

Otherwise the **`api`** (and Alembic, which loads the same settings in **`env.py`**) will fail validation on startup.

### 4. Day-to-day: run the stack and apply migrations

From the **repository root** (where **`docker-compose.yaml`** lives), with **`.env`** present for **`POSTGRES_*`**, **`REDIS_PASSWORD`**, **`JWT_SECRET_KEY`**, etc.:

```bash
# Start database, Redis, and API (migrate service is NOT started automatically).
docker compose up -d

# Apply migrations to the Compose Postgres (DATABASE_URL uses host `database` inside the network).
docker compose --profile migrate run --rm migrate
```

Alternative (same image, override command). Prefer **`python -m alembic`** inside Docker:

```bash
docker compose run --rm api python -m alembic upgrade head
```

If you see **`exec: "alembic": executable file not found in $PATH`**, the image was built **before** Alembic was added to **`requirements.txt`**, or you are using a stale layer — run **`docker compose build --no-cache api`** and try again.

After you add or change files under **`alembic/versions/`**, **rebuild** so the new revision exists in the image:

```bash
docker compose build api && docker compose --profile migrate run --rm migrate
```

**Autogenerate** new revisions is still usually done **on the host** (§4) with **`DATABASE_URL=...@localhost:POSTGRES_PUBLISH_PORT...`** so you can edit Python files in the repo without rebuilding first; then commit **`alembic/versions/*.py`**, rebuild, and run **`migrate`** in Compose.

### 5. Plain `docker compose up` does not run migrations

That is intentional: schema changes stay an explicit step after backup/review in real deployments. For local dev you can run **`migrate`** whenever the DB is empty or behind.

## Dev vs production (read this first)

**The commands are mostly the same** (`alembic revision …`, `alembic upgrade head`). What changes is **which database** they touch: whatever **`DATABASE_URL`** resolves to when Alembic runs.

|                            | **§4 — Developers (day-to-day)**                                                        | **§5 — Production**                                                                                                                                                                                |
| -------------------------- | --------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Goal**                   | Create migration files and update your **development** DB while you build features.     | Apply **already reviewed** migrations to the **live** (or staging) DB safely.                                                                                                                      |
| **Where you run it**       | Your **laptop**: repo root, usually with **`.venv`** activated.                         | **Not** your everyday dev shell unless you are deliberately pointing at prod (rare). Usually: **CI/CD job**, **SSH on a server**, **hosted runner**, or a **one-off container** with prod secrets. |
| **Typical `DATABASE_URL`** | `localhost` + host port (e.g. `5435`) — your Docker Postgres **published** to the host. | Hostname + credentials from your **hosting provider** (RDS, managed Postgres, VM on private network, etc.).                                                                                        |
| **Updates the DB?**        | Yes — **`alembic upgrade head`** changes **only** that dev database.                    | Yes — **`alembic upgrade head`** changes **only** the DB behind **that** URL (often production).                                                                                                   |

So: **§4 does not “only edit files.”** `alembic upgrade head` **always** applies migrations to the database in **`DATABASE_URL`**. In §4 you intend that URL to be **dev**. In §5 you set it to **production** (or run in an environment where it is already set to production).

## 4. Day-to-day workflow (developers — your machine → development database)

### Where to run the terminal (folder + virtualenv)

1. **Current directory = project (repository) root** — the folder that contains **`alembic.ini`** next to **`app/`** and **`docker-compose.yaml`** (for this repo, that is `fastapi-postgresql-crud/`, not `app/` and not `alembic/`). Alembic resolves paths from that root; if you `cd` into a subfolder, commands often fail with “can’t find alembic.ini” or wrong imports.

   ```bash
   cd /path/to/fastapi-postgresql-crud
   pwd   # should show the directory that contains alembic.ini
   ```

2. **Virtual environment (`.venv`)** — use it if you installed Alembic with **`pip install alembic`** inside that project’s venv (recommended so the Alembic version matches your `requirements.txt`).

   ```bash
   . .venv/bin/activate # Linux / macOS
   # or: .venv\Scripts\activate   # Windows cmd
   ```

   After activation, `which alembic` (Linux/macOS) or `where alembic` (Windows) should point **inside** `.venv`. You are not “inside” the `.venv` folder as your cwd; you stay at the **repo root** and only **activate** the venv in that shell.

   If Alembic is installed globally and you skip the venv, commands can still work, but you may get version mismatches—prefer **repo root + activated `.venv`**.

3. **`DATABASE_URL`** — your shell (or `.env` at the repo root, read by `get_settings()` in `alembic/env.py`) must point at the **development database** you want to migrate. For Postgres in Docker on the host, that is usually **`localhost`** and your **published** port (e.g. `5435`), not `database:5432` (that hostname only works **inside** the Compose network).

**Create a migration** after you change SQLAlchemy models (on a dev DB that reflects the previous migration state):

```bash
# Repo root, venv activated (recommended), DATABASE_URL → dev DB.
alembic revision --autogenerate -m "add items description length or new table"
```

**Always open the new file under `alembic/versions/`** and **review** the generated `upgrade()` / `downgrade()`. Autogenerate can miss renames, custom types, or partial changes.

**Apply** migrations to the **development** database (the one in your `.env` / `DATABASE_URL`):

```bash
alembic upgrade head
```

This **does** change the database — on purpose, so your dev DB matches your models. It does **not** touch production unless your `DATABASE_URL` is wrongly set to production (don’t do that on your laptop).

**Inspect** current revision:

```bash
alembic current
```

**History:**

```bash
alembic history
```

## 5. Running migrations in production (trusted environment → production database)

Use this when you want **`alembic upgrade head`** to run against the **production** (or **staging**) Postgres — **not** the `localhost:5435` dev DB from §4.

### Where you run it (examples)

Pick **one** pattern your team trusts; the common theme is “a machine or job that has **production** `DATABASE_URL` and is allowed to change the live schema.”

| Where                                                       | When it fits                                                                                                                        |
| ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **CI/CD pipeline** (GitHub Actions, GitLab CI, etc.)        | Step before or after deploy; secrets store `DATABASE_URL`.                                                                          |
| **Application server** (SSH into the box that runs the API) | You `cd` to the release directory (or clone), export prod URL, run `alembic upgrade head`.                                          |
| **One-off Docker Compose / Kubernetes job**                 | Image contains the repo + Alembic; job receives prod URL as an env var; command is `alembic upgrade head`.                          |
| **Your laptop**                                             | Only if you **manually** set `DATABASE_URL` to production and your team approves — easy to get wrong; many teams **disallow** this. |

You are **not** required to use a different Alembic command in production — same **`alembic upgrade head`**. The difference from §4 is **environment** (which host, which secrets) and **process** (backup, review, deploy order).

### Before deploy

1. **Backup** Postgres (snapshot, `pg_dump`, or your host’s backup tool).
2. Run migrations from one of the **places above**, with **`DATABASE_URL`** set to **production** (or staging).
3. **Review** the migration SQL if the change is large or high-risk (`alembic upgrade head --sql` can print SQL without applying, depending on setup).

### Typical command

On the **production runner** (after secrets are loaded):

```bash
export DATABASE_URL="postgresql+psycopg://..."  # production URL from secrets — NOT your dev .env on localhost unless this IS prod
alembic upgrade head
```

If the runner has a `.env` or injected env vars that already set `DATABASE_URL` for production, you may only need `alembic upgrade head` (no manual `export`).

### Order relative to the API

- **Common pattern:** run **`alembic upgrade head`**, then **deploy** the new API version that expects the new schema (or deploy only after migrations succeed).
- Avoid running a **new** app against an **old** schema, or an **old** app against a **breaking** new schema, without a coordinated rollout.

### Docker Compose (reference)

The **full** walkthrough (why each change exists, rebuild rules, `Settings` sync) is in **§ Docker Compose: set up migrations and what to change** near the start of this document. Below is a **clean copy-paste reference** for the two files and the commands to run after editing.

#### Step 1 — `Dockerfile`

Add Alembic artifacts and `PYTHONPATH` **after** `COPY app ./app` so the image matches [Dockerfile](../Dockerfile):

```dockerfile
FROM python:3.12-slim

WORKDIR /app

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY app ./app
COPY alembic.ini .
COPY alembic ./alembic

ENV PYTHONPATH=/app

EXPOSE 8000

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

#### Step 2 — `docker-compose.yaml`

1. Define a **YAML anchor** at the **top** of the file (Compose extension keys may start with `x-`) with every variable [app/config.py](../app/config.py) requires.
2. Point **`api.environment`** at it with **`<<: *api-environment`**.
3. Add a **`migrate`** service with the **same** environment, **`profiles: ["migrate"]`**, and **`command: python -m alembic upgrade head`**.

Example aligned with [docker-compose.yaml](../docker-compose.yaml):

```yaml
x-api-environment: &api-environment
  DATABASE_URL: postgresql+psycopg://${POSTGRES_USER}:${POSTGRES_PASSWORD}@database:5432/${POSTGRES_DB}
  REDIS_URL: redis://:${REDIS_PASSWORD}@redis:6379/0
  APP_NAME: ${APP_NAME:-FastAPI PostgreSQL CRUD}
  ENVIRONMENT: ${ENVIRONMENT:-development}
  DEBUG: ${DEBUG:-false}
  JWT_SECRET_KEY: ${JWT_SECRET_KEY:-change-me}
  JWT_ALGORITHM: ${JWT_ALGORITHM:-HS256}
  ACCESS_TOKEN_EXPIRE_MINUTES: ${ACCESS_TOKEN_EXPIRE_MINUTES:-30}
  REFRESH_TOKEN_EXPIRE_DAYS: ${REFRESH_TOKEN_EXPIRE_DAYS:-7}

services:
  api:
    build: .
    environment:
      <<: *api-environment
    # ... ports, depends_on, etc.

  migrate:
    profiles: ["migrate"]
    build: .
    environment:
      <<: *api-environment
    command: python -m alembic upgrade head
    depends_on:
      - database
```

Keep your existing **`database`**, **`redis`**, **`volumes`**, and **`api`** `ports` / **`depends_on`** as in the repo; the snippet shows only the **pattern** for env + **`migrate`**.

#### Step 3 — Run migrations in Compose

From the **repository root** (where **`docker-compose.yaml`** and **`.env`** live):

1. **Ensure `.env`** defines at least what Compose substitutes: `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`, `REDIS_PASSWORD`, and any JWT / `APP_NAME` overrides you use (see [.env.example](../.env.example)).
2. **Start** Postgres (and Redis/API if you need them):
   `docker compose up -d`
3. **Rebuild** the API image after Dockerfile or `alembic/versions/` changes so the container has the latest migrations:
   `docker compose build api`
4. **Inspect before migrating** (recommended): see what the DB has vs what the code expects.

   | Command | What it shows |
   | ------- | ------------- |
   | `python -m alembic current` | Revision **stored in the database** (last applied migration), or empty if Alembic has never run on this DB. |
   | `python -m alembic heads` | **Latest revision(s)** in your migration scripts inside the image (the target of `upgrade head`). |
   | `python -m alembic history -r current:head` | **Pending** revisions from DB state to branch head—what `upgrade head` would still apply. If output is empty and `current` equals `heads`, you are already up to date. |

   Prefix each with Compose (same as other one-off commands):

   ```bash
   docker compose run --rm api python -m alembic current
   docker compose run --rm api python -m alembic heads
   docker compose run --rm api python -m alembic history -r current:head
   ```

   If **`current`** prints nothing (no `alembic_version` row yet), list everything that would run up to head with:

   ```bash
   docker compose run --rm api python -m alembic history -r base:head
   ```

5. **Apply** migrations (pick one):
   - `docker compose --profile migrate run --rm migrate`
   - `docker compose run --rm api python -m alembic upgrade head`
6. **Verify after migrating** (optional but useful):

   ```bash
   docker compose run --rm api python -m alembic current
   ```

   It should match **`alembic heads`** from step 4.

`migrate` is **not** started by plain `docker compose up`; run step 5 when you want the schema updated.

For **production** Compose stacks, use the same commands but ensure **`x-api-environment`** (or secrets) points at the **correct** database—never run a prod **`migrate`** with a dev `.env` by mistake.

## 6. Existing database already created with `create_all`

If development (or an early deploy) created tables with **`create_all`** and you **introduce Alembic later**:

- **Option A — Fresh dev DB:** drop dev data, run **`alembic upgrade head`** from the first revision, then continue.
- **Option B — Keep data:** generate an initial migration that matches the **current** database, apply **`alembic stamp head`** once to mark the DB as up-to-date without re-running `CREATE TABLE`, or hand-edit the first revision. This is easy to get wrong; prefer **Option A** for learning; for production, involve someone comfortable with Alembic **stamp** vs **upgrade**.

When in doubt, practice on a **copy** of the database first.

---

## 7. Checklist (production)

- [ ] **`ENVIRONMENT`** is not `development` on production servers so **`init_development_tables()`** does not run.
- [ ] **`alembic upgrade head`** is automated or documented; secrets (`DATABASE_URL`) are not logged.
- [ ] Migrations are **reviewed** in pull requests; destructive steps include a **backup / rollback plan**.
- [ ] **CI** runs migrations against a **staging** database that mirrors production constraints before production apply.
